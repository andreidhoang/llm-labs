## Tier 2 promotion candidates

### → H₄ (sweep_design.md, cell E5: dense vs MoE) — STRONG support

The most load-bearing finding of the session. At d=8, 5-min budget:

- **MoE-on (iter2):** 1.105, 476 steps, 125M tokens
- **Dense (iter4):** 1.063, 725 steps, 190M tokens (+52% throughput)
- **MoE-on with MORE experts (iter3, NUM_EXPERTS=8):** 1.142 (regression)

The per-step capability of MoE cannot overcome the wall-clock throughput penalty
at this scale (fewer total tokens seen). This directly replicates nanochat's
d=24 H₄ finding (per-step win, wall-clock loss) at a smaller scale and shorter
budget — strong cross-scale evidence.

**Recommended action for Tier 2:** keep E5 (dense G=1) cell in sweep_design.md
high-priority. Strong small-scale prior says dense ships faster wall-clock to T.
Consider also adding a G=4 cell (between current G=2 baseline and dense E5) to
characterize the curve, since iter3's NUM_EXPERTS=8 result hints the
relationship may be non-monotonic.

### → LR envelope shift at smaller depths (not directly preregistered)

Karpathy-anchored HPs (MATRIX_LR=0.04, EMBED_LR=0.6) beat llm-labs's
conservative nanochat-d24-anchored defaults (0.02, 0.3) at d=8:

- baseline (LR=0.02, 0.3): 1.121
- iter1 (LR=0.04, 0.3): 1.109 (−0.012)
- iter2 (LR=0.04, 0.6): 1.105 (−0.016 cumulative)

The 1/√(d_model) scaling rule used in `core/model.py:setup_optimizer` is
calibrated at d=24. At d=8 the implied LRs are too conservative — Karpathy's
higher absolute LRs win by ~1.5%. **Recommended action:** consider adding a
mini-LR-sweep cell to Tier 2 if any Tier 2 cell drops below d=24 (none currently
planned, so this is a deferred finding).

## Port fidelity verification (T5 result)

The anchor experiments (dense, Karpathy-exact HPs, 50.3M params) landed at
val_bpb 1.081 vs Karpathy's published 0.998 — a gap of 0.083, with:

- **Parameter count exactly matched** (50.3M vs 50.3M).
- **Same architecture** (dense GPT, RoPE, QK-norm, value embeddings, ResFormer,
  Muon+AdamW, ReLU² FFN).
- **Same metric** (`core/loss_eval.evaluate_bpb`, vocab=8192, climbmix val
  shard, EVAL_TOKENS=40·524288).

The gap is explained by **throughput**, not numerical correctness:

| | Our run (H200) | Karpathy (H100 published) |
|---|---|---|
| total_tokens_M (5 min) | 208 | 500 |
| MFU | 17% | 40% |
| Flash Attention | FA2 (pip flash-attn 2.7.3) | FA3 (sm_90 HF hub kernel) |

H200 is sm_90 Hopper (capability 9,0) — FA3 is available in principle but
requires the `kernels` package + HF Hub kernel pull. NGC 25.03 image doesn't
ship `kernels` by default; installing it adds ~30s but should close most of
the throughput gap.

**Conclusion: port is faithful.** Bit-level differences are infrastructure
(FA2 vs FA3 + H200 vs H100 instruction mix), not algorithm.

## What worked in the loop

1. **Karpathy's primitive scales to llm-labs's core/.** Agent edits `core/`
   directly (not a fork), branch isolation keeps Tier 2 safe. Branch advances
   on win, resets on loss — visible in git history.
2. **`grep "^val_bpb:" run.log`** is the only orchestration needed. No
   launcher, no differ, no proposer — the loop runs on git + grep.
3. **Wall-clock budget makes changes commensurable.** iter3 (NUM_EXPERTS=8)
   ran fewer steps (354) because MoE is heavier; the system correctly punished
   that with a higher val_bpb.

## What to improve

1. **Install `kernels` package** in the prepared image (or `pip install
   kernels` in `auto/README.md` quickstart) so FA3 is available on Hopper.
   Closes most of the 0.083 gap to Karpathy's anchor.
2. **MFU at d=8 is low** (11-19% on MoE-on, 17% on dense+SSSL). Acceptable for
   research iteration but suggests a `torch.compile` regression or kernel
   overhead. Profile in a future session if MFU becomes load-bearing.
3. **Session-marker rows in results.tsv** (the `3507121` row with val_bpb 0)
   work but are clunky. Consider a `notes:` column or a separate `branches.tsv`
   for session organization in future programs.md iterations.

## Reproducibility

Branch `auto/2026-05-17` is preserved on the Vast instance. To reproduce on a
fresh 1×H200:

```bash
git checkout auto/2026-05-17 -- auto/train_auto.py    # any commit on the branch
python auto/prepare_auto.py                            # one-time
python auto/train_auto.py > run.log 2>&1               # 5 min
grep "^val_bpb:" run.log
```

Each commit on the branch is one experiment; the diff `git show <SHA> --
auto/train_auto.py` shows exactly what changed.
