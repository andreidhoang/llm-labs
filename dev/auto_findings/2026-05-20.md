## Findings

### Finding 1 — FA3 source-build path is too costly for Tier 1

**Attempted:** clone Dao-AILab/flash-attention, run `python setup.py install` in `hopper/` subdir on NGC pytorch:25.03-py3 (has nvcc; CPU compile).

**Result:** abandoned after 45 minutes (CPU-bound ptxas compilation of ~100+ sm_90 kernel specializations: head_dim ∈ {64,128,192,256} × dtype ∈ {fp16,bf16,fp8} × variants ∈ {paged, packgqa, softcap, split}). Build log reached 46K lines, GPU at 0% utilization (entirely CPU work).

**Estimated total build time:** 60-90 minutes on this host. Cost: ~$4-6 just to compile, before running any experiment.

**Conclusion:** FA3 from-source build path INCOMPATIBLE with autoresearch's "cheap experiments" model. Karpathy's primitive assumes setup overhead < experiment cost; here it's 10× larger.

**Options for future FA3 sessions:**
1. **Pre-built wheels** — wait for Dao-AILab to publish wheels matching NGC torch 2.7.0a0+nv25.03 ABI. Currently they don't.
2. **Pre-built Docker image** — build FA3 once on a -devel image, persist as image layer, reuse for autoresearch instances. Setup is one-time engineering cost.
3. **Different upstream kernel** — find a `kernels-community/*` repo that's both NGC-ABI-compatible AND publicly readable. Sessions 3+4 tested 3 candidates, all failed.
4. **Accept FA2 ceiling** — Tier 1 stays at FA2 ~195M tokens/5min on H100. Tier 2 (NGC -devel image) gets FA3.

**Recommended:** option 2 (build FA3 into a custom Docker image). One-time ~1-hour engineering investment; subsequent autoresearch sessions get FA3 for free.

### Finding 2 — FP8 hurts at d=8/50M params; helps at d=10+/80M+

**Per-scale results with FP8 enabled via `core.fp8.convert_to_float8_training`:**

| DEPTH | params | val_bpb | tokens (M) | MFU | Notes |
|---|---|---|---|---|---|
| 8 (FP8 off) | 50M | 1.084 | 144 | 11.6% | iter1 BF16 baseline |
| 8 (FP8 on) | 50M | 1.090 | 138 | 11.1% | **WORSE** — quant overhead > compute savings |
| 10 (FP8 on) | 80M | 1.080 | 118 | 17.1% | session 4 best |
| 12 (FP8 on) | 120M | 1.085 | 95 | 22.1% | undertrained at 5min budget |

**Mechanism:** FP8 quantize/dequantize wrappers around each matmul add per-step overhead. At d=8, model is small enough that compute (the part FP8 speeds up) is dominated by Python/CUDA dispatch overhead (which FP8 makes WORSE due to extra ops). MFU goes DOWN with FP8 at d=8.

At d=10+, model is large enough that compute starts to dominate per-step time. MFU jumps from 11% (d=8 BF16) to 17-22% (d=10-12 FP8). The compute savings exceed the quant overhead, and FP8 net-helps.

**Sweet spot:** DEPTH=10, ~80M params, FP8 on. Marginal val_bpb win (−0.004 vs same-host FA2 baseline) and 50% higher MFU (matters for energy cost, not just val_bpb).

**Tier 2 implication:** sweep_design.md's d=24 cells should DEFINITELY use FP8 (sweep_design v3 already plans this — H₀ tests FP8-on-shared-expert speedup). Validated cross-scale: FP8 helps at d=10+, will help more at d=24.

### Finding 3 — Inter-host throughput variance is ~25%

iter1 today (host 37045648): 144M tokens, MFU 11.6% at d=8 BF16
session 3 iter4 (host 36997785): 195M tokens, MFU ~13% at d=8 BF16

Same Vast.ai 1×H200 spec, same NGC image, same code, same hyperparameters — different physical host gives 35% more tokens. Implications:

- **Absolute val_bpb across sessions is contaminated** by host variance
- **Within-session deltas are reliable** (same host throughout)
- For Tier 2: pin to one Vast.ai instance for full sweep, OR accept inter-cell variance and use bootstrap CI

This validates sweep_design.md's "single-instance discipline is the most important infra rule" (§11.1).
